# Tiny INT8 inference firmware

This firmware mode is a bridge between the synthetic instruction benchmarks
and a future TensorFlow Lite Micro port. It does not use TFLM yet. Instead, it
runs one small exported fixed-shape INT8 model directly in bare-metal C:

```text
8x8 input image -> 16 hidden units -> 4 output classes
```

The current exported model is an `8x8` quadrant classifier. Its firmware
artifact lives in `tiny_ai_model.h` and is generated by:

```bash
cd /home/duydonv/de2i150_cv32e40p_soc/firmware
python3 export_tiny_ai_model.py
```

The baseline path uses scalar RV32IM C loops. The custom path computes each
fully-connected dot product with the existing CORE-V/PULP wrappers:

- `cv.sdotsp.b` for packed signed INT8 dot products
- `cv.lw` post-increment loads for activation and weight streams
- `cv.setup` for the inner dot-product hardware loop
- `cv.clipur` for ReLU-style clamp after requantization

Build it with:

```bash
cd /home/duydonv/de2i150_cv32e40p_soc/firmware
make tiny_ai
```

After the firmware is rebuilt, re-run the Quartus compile so the top-level
BRAM initialization picks up the new `firmware_byte{0..3}.hex` files.

The UART report prints the baseline/custom cycle counts, instruction counts,
cycles per sample, cycles per INT8 MAC, checksums, predicted classes, and the
first sample's output scores. A pass means the scalar and custom inference
paths produced identical scores and classes for all fixed samples.

The exported sample labels are:

```text
0 0 1 1 2 2 3 3
```

The model performs `8 * ((16 * 64) + (4 * 16)) = 8704` signed INT8 MACs.
The custom path groups those MACs into `2176` packed dot4 operations.
The expected checksum for the exported samples is `0x6c75e8cf`.

Latest observed exported-model board output on the DE2i-150:

| Variant | Cycles | Instret | Cycles/sample | Cycles/MAC | Checksum | Pass |
|---|---:|---:|---:|---:|---|---|
| baseline | 91,106 | 63,917 | 11,388 | 10.47 | `0x6c75e8cf` | yes |
| custom | 12,848 | 9,369 | 1,606 | 1.48 | `0x6c75e8cf` | yes |

Observed speedup: `7.09x`. Accuracy is `8/8`, with matching baseline/custom
classes:

```text
0 0 1 1 2 2 3 3
```

The first bring-up version used pseudo-random generated weights and observed
`7.10x` speedup on the DE2i-150 (`91,237` baseline cycles vs. `12,859` custom
cycles). The exported-model version above is the current pre-TFLM reference.

LED markers are compatible with the existing power testbench:

- `0x40` marks the start of the measured inference work.
- `0xa5` marks scalar/custom checksum pass.
- `0xef` marks scalar/custom checksum mismatch.

To build one variant for the existing power testbench, use `POWER_SIM`:

```bash
make PROGRAM=tiny_ai EXTRA_CFLAGS="-DPOWER_SIM -DPOWER_SIM_VARIANT=0" clean all
make PROGRAM=tiny_ai EXTRA_CFLAGS="-DPOWER_SIM -DPOWER_SIM_VARIANT=1" clean all
```

`0` selects the scalar baseline path and `1` selects the CORE-V/PULP custom
path.

This is intentionally close to a future optimized TFLM `FullyConnected` kernel:
once a small TFLM model runs with the reference kernels, the inner dot-product
body here can be moved into the TFLM operator implementation or used as a
custom-op backend.
